# Investment Research RAG System

A retrieval-augmented generation (RAG) system for investment documents.  
Ask natural-language questions over multiple PDFs and receive grounded answers with page-level citations.

---

                                                     │
                              ┌──────────────────────┼──────────────────────┐

                     │  Supabase       │  │  PageIndex     │  │  Claude API        │

### Component roles

| Component | Role |
|-----------|------|
| **Streamlit** | Two-page UI: chat interface + document management |
| **FastAPI** | REST API layer; runs background indexing workers |
| **PageIndex Cloud** | PDF parsing → hierarchical tree index; per-document Q&A |
| **Claude (Anthropic)** | Doc relevance selection (Haiku) + multi-doc synthesis (Sonnet) |
| **Supabase PostgreSQL** | Persistent storage of chat history and indexing job state |

## Database Design

### `chat_history`
| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID PK | auto-generated |
| `question` | TEXT | user question |
| `answer` | TEXT | generated answer |
| `sources` | JSONB | `[{"doc_name", "doc_id", "pages"}, …]` |
| `created_at` | TIMESTAMPTZ | |

### `indexing_jobs`
| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID PK | auto-generated |
| `filename` | TEXT | original PDF name |
| `file_path` | TEXT | local path |
| `status` | TEXT | `pending` → `processing` → `completed` \| `failed` |
| `pageindex_doc_id` | TEXT | `pi-xxx…` returned by PageIndex |
| `doc_description` | TEXT | auto-generated by PageIndex |
| `page_count` | TEXT | total pages |
| `error_message` | TEXT | set on failure |
| `created_at` / `updated_at` | TIMESTAMPTZ | |

Tables are created automatically on first server start via `create_tables()`.

## Retrieval Approach (vectorless RAG)

PageIndex builds a **hierarchical tree index** of each PDF (like a semantic table of contents).  
At query time, it uses LLM reasoning to traverse the tree and fetch only the relevant pages.

This avoids the similarity ≠ relevance problem of traditional vector search, giving more precise
and traceable retrieval without a vector database.

### Multi-document pipeline

```
question
   │
   ▼
Claude Haiku — select relevant docs (using only filenames + descriptions)
   │
   ▼  (parallel per relevant doc)
PageIndex chat_completions → per-doc answer + page citations
   │
   ▼
Claude Sonnet — synthesise final answer with cross-doc attribution
   │
   ▼
{ answer, sources: [{ doc_name, pages }] }
```

## Handling Key Challenges

### Version awareness
When multiple versions of the same company's materials are uploaded (e.g. Q1 2023 deck and Q3 2024 deck),
the synthesis prompt explicitly instructs Claude to **treat each document separately** and note the time period.
Documents are never blindly merged; values are always attributed to their source.

### Conflicting information
The Claude synthesis prompt mandates: *"If values or facts conflict across documents, surface the conflict
explicitly rather than picking one silently."*  
Example output: *"The 2022 deck reported ARR of $45M [Deck 2022, pp. 8], while the 2024 update reported
$72M [Deck 2024, pp. 5]."*

### Charts and tables
PageIndex extracts the PDF **text layer** only.  Numeric data embedded in text tables is usually captured.
Figures and chart images are not extracted; when the answer mentions a chart that couldn't be read,
the synthesis prompt instructs Claude to note: *"(chart data not extractable from PDF text layer)"*.

## Setup Instructions

### 1. Prerequisites

- Python 3.11+
- A free Supabase project (or any PostgreSQL instance)
- A [PageIndex API key](https://dash.pageindex.ai/api-keys)
- An Anthropic API key

### 2. Environment variables

Edit `../.env` (the root `.env`, one level above this directory):

```bash
ANTHROPIC_API_KEY=sk-ant-...
PAGEINDEX_API_KEY=your_pageindex_key
DATABASE_URL=postgresql://user:pass@host:5432/dbname
```

### 3. Install dependencies

```bash
cd app
pip install -r requirements.txt
```

### 4. Run

**Linux / macOS:**
```bash
bash run.sh
```

**Windows:**
```bat
run.bat
```

Or start each server manually in separate terminals:

```bash
# Terminal 1 — backend (from app/ directory)
uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload

# Terminal 2 — frontend (from app/ directory)
streamlit run frontend/Home.py --server.port 8501
```

Open **http://localhost:8501** in your browser.

## Usage

1. Go to the **Documents** page and upload your PDFs.
2. Wait for indexing to complete (1–5 min per document; auto-refresh shown).
3. Return to the **Chat** page and ask questions.
4. Each answer shows **Sources** with document name and page range.

## Known Limitations

| Limitation | Mitigation / Notes |
|------------|-------------------|
| Charts / images not extracted | Noted in answers; text-layer tables usually work |
| Indexing is asynchronous | Progress shown via polling; typically 1–5 min |
| Scanned PDFs (no text layer) | May index poorly; use OCR-processed PDFs |
| Very large PDFs (> 200 pp.) | May take longer; PageIndex handles gracefully |
| Per-document citation granularity | Citations reference page ranges, not exact sentences |
| No user authentication | All documents visible to all users (see access control note below) |

---

## Access Control (Optional Consideration)

The current system is single-tenant.  To support multiple clients:

1. **DB**: Add a `client_id` column to `indexing_jobs`.  All queries filter by `client_id`.
2. **Auth**: Add JWT middleware to FastAPI; embed `client_id` in the token.
3. **PageIndex**: Store a per-client `pageindex_doc_id` mapping; queries only access that client's docs.
4. **UI**: Pass an API token from Streamlit; the backend validates and scopes document access.
